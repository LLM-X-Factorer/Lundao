# Lundao — 产品定位

## 一句话

**面向中文学术研究者的论文汇报自动化工具，三分钟把一篇论文变成可讲的 PPT、讲稿、技术文章。**

## 目标用户

按重要度排序：

1. **中文研究生 / 博士生**（终端用户，第一优先级）
   - 痛点：每周组会要讲 1-2 篇论文，从读到讲耗时 3-5 小时
   - 期待：上传 PDF 或选一篇 arXiv → 喝杯咖啡的时间拿到讲材料
   - 行为：用得多、付费意愿低，需要"开箱即用"

2. **开发者 / 贡献者**（开源协作者，第二优先级）
   - 痛点：想学/改 AI 多 agent 工作流，找不到既完整又看得懂的开源项目
   - 期待：清晰的架构文档、可替换的组件（换 LLM、换 PPT 引擎）、低门槛贡献流程
   - 行为：fork、改造、提 PR

3. **学习者**（教学受众，第三优先级）
   - 关心代码可读性、注释、教学性的架构 doc
   - 把这个项目当 LangGraph / 多 agent / Vue 3 现代实践的 reference

## 价值主张

为什么不直接用 ChatGPT / Gemini Web 端？

| 维度 | ChatGPT/Gemini 直接用 | Lundao |
|---|---|---|
| 上手成本 | 自己写 prompt、自己处理论文格式 | 一键，prompt 已工程化 |
| 中文输出质量 | 一般，需多轮修 | 专为中文学术汇报调过的 5 个 prompt |
| 多产物 | 一次只能聊一类 | 一次出 4 件（PPT 蓝图 / 深度解读 / 技术文章 / 演讲稿）+ PPT 文件 |
| 多模态 | 需手动贴图 | 自动起本地图床，把论文里的图喂给模型 |
| 可复制性 | 临时对话 | 输出保存在本地，可分享、可二次编辑 |

**核心壁垒不是模型，是 prompt + 工作流 + 产物组合**。

## 明确不做的事

为了保持工具属性，划清边界——以下事**不做**：

- ❌ **用户系统**：不注册、不登录、不存账号。任务状态用浏览器 localStorage + 后端 SQLite（按 taskId 取）
- ❌ **付费 / 订阅**：用户带自己的 API key，本地跑或自部署。我们不做 SaaS
- ❌ **社交 / 推荐 / 评分**：不是社区，不是平台
- ❌ **多语种 UI**：中文学术场景优先（界面可英文，但 prompt 中文是 first-class）
- ❌ **手机端深度优化**：研究生汇报场景在 PC 桌面，移动端只保证不崩

任何引入这些会改变工具属性的 feature request，会在 issue 里直接 close 并指向本文档。

## 成功指标

衡量"做好了"的具体定义：

- **用户**：一个研究生不需要看文档就能用，上传论文到拿到 PPT 草稿 ≤ 3 分钟
- **开发者**：从 `git clone` 到本地跑通 ≤ 10 分钟（前提是有 Gemini API key）
- **学习者**：架构文档能让一个会 Python 但不会 LangGraph 的人 30 分钟内看懂整个工作流
