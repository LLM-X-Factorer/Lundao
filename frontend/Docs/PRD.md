## **“论导Lite” - MVP 前端需求规格说明书 (V1.0)**

### 1. 文档概述

#### 1.1. 项目背景与愿景
“论导Lite”是一款面向中国科研学生群体的SaaS工具。项目愿景是利用AI技术，将“筛选论文 -> 理解论文 -> 汇报论文”的流程极致简化，让用户以最快速度、最低心智负担完成组会PPT的准备工作。本项目为该SaaS的最小可行产品（MVP）的前端部分，**严格遵循“工具而非平台”的原则**。

#### 1.2. 目标用户
中国的硕士、博士研究生，以及需要定期进行学术文献汇报的本科生。

#### 1.3. 核心设计原则
* **工具化，非平台化 (Tool-oriented, not Platform-oriented)**：所有功能都为“用完即走”的核心任务服务，不做任何增加用户留存负担的“管理”功能。
* **单页极简 (Single-Page Minimalism)**：核心功能全部承载于单一页面，避免任何不必要的页面跳转和导航。
* **零摩擦体验 (Frictionless Experience)**：**无需注册/登录**即可使用全部核心功能。
* **价值前置 (Value First)**：将AI分析的核心结果（中文摘要、创新点）在第一时间、最显眼的位置展示给用户。
* **美学优先 (Aesthetics First)**：UI设计需现代、简洁、富有科技感，通过优秀的视觉设计和动效提升产品的专业度和信任感。

### 2. 功能模块详细说明

#### 2.1. 整体页面布局 (Single Page Layout)
整个应用为一个单页应用，所有元素均在根路由 (`/`) 下。
* **顶部导航栏 (Header)**：
    * 内容：仅包含产品Logo和Slogan（例如：“论导Lite - AI助你三分钟搞定组会PPT”）。
    * 设计：固定在页面顶部，视觉上轻量，不占用过多空间。
* **核心操作区 (Main Section)**：
    * 页面的视觉中心，分为左右或上下两个主要功能模块。
* **任务历史区 (History Section)**：
    * 位于核心操作区下方，是用户操作结果的反馈区域。
* **页脚 (Footer)**：
    * 包含产品简介、反馈渠道、服务条款等次要链接。

#### 2.2. 模块一: 核心操作区

##### 2.2.1. 功能A: 发现前沿论文 (Discover Papers)
* **组件构成**:
    1.  **时间维度Tabs**: 提供“今日热门”、“本周热门”、“本月热门”三个可切换的标签页。
    2.  **论文卡片列表**:
        * 每个Tab下为一个论文列表，通过卡片(Card)形式展示。
        * 卡片信息：论文标题 (Title)、主要作者 (Authors)、所属领域/关键词 (Keywords/Category)。
        * 交互：鼠标悬浮(Hover)在卡片上有视觉反馈（如轻微放大或阴影变化）。
    3.  **分页组件 (Pagination)**: 列表底部提供简洁的分页功能。
* **交互逻辑**:
    * 用户点击任意论文卡片，**触发“论文详情与操作弹窗”**。

##### 2.2.2. 功能B: 上传我的论文 (Upload My Paper)
* **组件构成**:
    * 一个设计精美的**文件拖拽上传区域 (File Dropzone)**。
* **组件状态**:
    1.  **默认状态**: 提示性文案，如“将PDF文件拖拽至此，或点击选择文件”。
    2.  **拖拽悬停状态**: 区域边框高亮，并显示提示图标，给用户明确的视觉反馈。
    3.  **上传中状态**: 显示上传进度条和百分比，禁止用户重复操作。
    4.  **成功/失败状态**: 上传完成后，给予短暂的成功（如打勾动画）或失败（如错误提示）反馈。
* **交互逻辑**:
    * 文件上传成功后，**自动触发“论文详情与操作弹窗”**，并加载该PDF的分析结果。

#### 2.3. 模块二: 论文详情与操作弹窗 (Paper Detail & Action Modal)
这是产品的交互核心，设计必须精良。
* **触发方式**: 点击论文卡片 或 上传PDF成功。
* **弹窗布局**:
    * **标题区**: 清晰展示论文完整标题。下方可附带作者、来源等信息。
    * **AI核心分析区**:
        * **中文摘要**: 作为一个独立的区块，内容直接全部展示。
        * **创新点提炼**: 使用项目符号列表 (Bulleted List) 形式，清晰地逐条罗列，每条创新点力求精炼。
        * **加载状态**: 在等待后端AI分析结果时，此区域需显示优雅的骨架屏 (Skeleton Screen) 或加载动画。
    * **核心行动点 (Call to Action - CTA)**:
        * 一个非常醒目、视觉突出的主按钮：**“一键生成组会PPT”**。
        * 按钮可设计交互状态：默认、悬浮、点击后加载中（如显示Spinner）。
    * **次要操作区**:
        * 在主按钮旁边或弹窗底部，以链接形式提供“[下载PDF原文]”、“[访问Arxiv页面]”等选项。
* **交互逻辑**:
    1.  用户点击“一键生成组会PPT”按钮。
    2.  按钮进入加载状态，向后端发送生成请求。
    3.  请求发送成功后，弹窗关闭。
    4.  页面弹出全局Toast提示：“PPT生成任务已创建，请在下方历史记录中查看状态”。
    5.  同时，页面自动在“任务历史区”新增一条任务记录，初始状态为“排队中”。

#### 2.4. 模块三: 任务生成历史 (Task Generation History)
* **数据存储**: **所有任务记录均存储于用户浏览器的 `localStorage` 中**，实现免登录状态保持。
* **UI展现**:
    * 一个任务列表，每一行代表一个任务。
    * 列表字段：论文标题、任务状态、创建时间、操作。
* **任务状态 (Status)**:
    * 必须有明确的视觉区分（如不同颜色的徽章 Badge）。
    * 状态定义：`排队中`、`生成中`、`已完成`、`失败`。
    * 状态更新：前端通过**定时轮询 (Polling)** 机制，每隔一定时间（如5秒）向后端查询任务状态并更新UI。
* **交互逻辑**:
    * 当任务状态变为`已完成`时，“操作”区域的按钮变为可点击的“**下载PPT**”按钮。
    * 当任务状态变为`失败`时，“操作”区域可显示“重试”按钮和失败原因提示。
    * 每条记录提供一个“删除”按钮，用于从`localStorage`中清除该条记录。

### 3. 非功能性需求

#### 3.1. UI/UX 设计要求
* **视觉风格**: 现代、简洁、专业，可采用深色模式或具有科技感的亮色模式。
* **动效**: 所有交互过程应有平滑、无打扰的过渡动画，如弹窗的淡入淡出、按钮的状态变化、列表项的新增等，提升产品质感。
* **响应式设计**: 优先为桌面端大屏幕优化，但需保证在常见笔记本电脑屏幕（如13-15英寸）上布局依然和谐，内容不会错乱。
* **空状态设计**: 当Arxiv列表加载失败或任务历史为空时，需要有友好的空状态提示页面。

#### 3.2. 性能要求
* **首屏加载**: 优化资源大小，利用代码分割等技术，确保首次访问速度。
* **交互响应**: 用户的任何点击操作都应在100毫秒内得到视觉反馈。

### 4. 前后端API交互约定

前端开发需要后端提供以下API接口：
1.  **获取Arxiv论文列表**:
    * `GET /api/arxiv_papers`
    * 参数: `period` (daily, weekly, monthly), `page`, `limit`
2.  **上传PDF文件**:
    * `POST /api/upload_pdf`
    * 返回: `file_id` (用于后续操作)
3.  **分析已上传的PDF/指定的Arxiv论文**:
    * `GET /api/analyze_paper`
    * 参数: `file_id` 或 `arxiv_id`
    * 返回: 论文的中文摘要、创新点等分析结果。
4.  **创建PPT生成任务**:
    * `POST /api/generate_ppt`
    * 参数: `file_id` 或 `arxiv_id`
    * 返回: `task_id`
5.  **查询任务状态**:
    * `GET /api/task_status`
    * 参数: `task_id`
    * 返回: 任务当前状态 (`status`) 和完成后的下载链接 (`download_url`)
